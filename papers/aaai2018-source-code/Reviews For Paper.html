- extends a known RNN approach to work on source code through a new GRU cell and beam search parameters
- provides a clever way of empirically evaluating the approach (google code jam sets)
- interesting use of RNN (with advances that make it better for this use case) and application
of RNN that addresses a hard problem, and having solutions that may be useful in some circumstances
- I would have liked to have seen more details on the insights/intuition as to the
various design and implementation decsions however (related to improving RNN)

weaknesses:`
- the empirical evaluation is weak (poor baseline choices, no comparison to related work (templates), and no user study is performed to show that the comments produced are at all useful to developers (vs reading code in high level languages which most do))
- unclear how important of a problem this is. As code has become much higher level in recent decades, much of it is understandable without comments so the 15% claim for 10 java projects lacks meaning in my opinion. I am unconvinced that reading code is harder than reading the comments generated here for complex methods especially for open source software systems (and no/fewer controls over contributor styles/standards). (suggestions related to this are below)
- unclear whether this will translate to other programming languages or programmer style guides (lacks generality)
- requires the source (would be significantly more interesting if you could extract semantics from Java bytecode imo) as source code is not always available and its much more challenging to understand a method in bytecode than in Java source

I suggest perhaps using this approach to see if developers in a company are using the company's required syle code (coding standard) which you may be able to infer/learn from the AST/naming conventions. I'd also suggest using the style guide as input or for part of the learning process (to save code review time which is very large and which every developer hates (both reviewee and reviewer)). Finally, evaluating or studying open source projects with a range of contibutors is more like the wild west than what is practiced in companies for proprietary source bases. There may be better results to be had by focusing in on more realistic and controlled settings.

minor:
- spell out and define acronyms upon first use (LDA, GRU) and cite
- please do not use contractions in a technical paper (doesn't, can't, don't, it's)

Question 
[Summary] Please summarize the main claims/contributions of the paper in your own words. The paper introduces a Recursive Neural Network specifically tailored for source code analysis (Code-RNN), which can synthesize in its output vector several aspects of the code syntax and semantics. Code-specific techniques, like and abbreviation CamelCase expansion, are also applied to unveil hidden semantics of identifiers. Code-RNN is initially used as a code classifier, then the authors introduce a Recurrent NN for the generation of code comments, with a GRU designed to take advantage of the output of the Code-RNN. The experiments show that the Code-RNN is an average code classifier and, more interestingly, that the Code-GRU based network, even with some intrinsic limitations, is able to generate code comments that clearly stand above the baselines.

[Significance] Are the results significant? Significant
[Novelty] Are the problems or approaches novel? Somewhat novel or somewhat incremental
[Soundness] Is the paper technically sound? Technically sound
[Evaluation] Are claims well-supported by theoretical analysis or experimental results? Very convincing
[Clarity] Is the paper well-organized and clearly written? Excellent
[Detailed Comments] Please elaborate on your assessments and provide constructive feedback. Comment generation is an "hot" issue in modern software projects, especially open source ones, since human-readable comments make complex functions immediately understandable to readers. I found the authors approach interesting and well compared with existing techniques. Much work has to be done in order to create comments comparable to those (not always) written by programmers, but this is an honest first step and the results are promising, so I think that the paper should be presented and discussed at AAAI.
